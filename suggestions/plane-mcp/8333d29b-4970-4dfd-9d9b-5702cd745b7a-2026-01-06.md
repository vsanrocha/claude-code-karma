## Pain Points & Issues

### 1. State Management is Confusing ❌

**Problem**:
```
list_work_items(expand="state")
→ Returns: state as OBJECT with id/name/group
→ But expansion fails with validation error!
```

**Error Encountered**:
```
4 validation errors for PaginatedWorkItemResponse
results.0.state
  Input should be a valid string [type=string_type,
  input_value={'id': '8e9ac283-c112-46e...', 'group': 'unstarted'},
  input_type=dict]
```

**Impact**:
- Could not expand state to see available statuses
- Had to hardcode state ID from previous query
- No way to know valid state transitions

**Suggested Fix**:
```python
# Option 1: Fix the Pydantic model to accept dict
class WorkItem:
    state: Union[str, dict]  # Allow both ID and expanded object

# Option 2: Add separate tool
list_work_item_states(project_id: str) -> List[State]

# Option 3: Return state info in work item by default
{
  "state": "94e4de76-c80e-4c6f-bcd2-5408c0b53a2c",
  "state_name": "In Progress",
  "state_group": "started"
}
```

**Priority**: 🔴 High - Blocks basic workflows

### 3. No Document/Page Access ❌ **CONFIRMED MISSING**

**Problem**: Work item mentioned "Look at pages in this repo", but pages are completely unsupported

**What I Tried**:
```python
list_documents(projectId=project_id)  # ❌ Tool doesn't exist

# Verified project has pages enabled
project = retrieve_project(project_id="ba9f6b13-3f7a-4e5b-94d8-c234b6243719")
assert project["page_view"] == True  # ✅ Pages ARE enabled in project

# Searched all 55 available MCP tools
# Result: ZERO tools for pages/documents
```

**Verification**:
- Total MCP tools available: **55**
- Tools for pages/documents: **0**
- Project has pages enabled: **Yes** (`page_view: true`)
- Conclusion: **Pages are completely unsupported in MCP server**

**Impact**:
- **Cannot access Plane pages/docs programmatically** - Must use web UI
- **Cannot read agent composition guidance** stored in Plane pages
- **Cannot programmatically document workflows** in Plane
- **Blocks knowledge management** use cases (wikis, runbooks, SOPs)
- **Forces context switching** between Claude and Plane web UI

**Real-World Example**:
```
Work Item: "Look at pages in this repo for agent composition patterns"
↓
Agent needs to read Plane page with architecture decisions
↓
❌ No MCP tool available
↓
User must manually copy-paste from Plane web UI
```

**Suggested Addition**:
```python
# Minimal page support
def list_project_pages(project_id: str) -> List[Page]:
    """List all pages in a project"""

def get_page(project_id: str, page_id: str) -> PageDetail:
    """Get page content (markdown preferred)"""

def search_pages(project_id: str, query: str) -> List[PageSearchResult]:
    """Full-text search across pages"""

# Nice-to-have
def create_page(project_id: str, title: str, content: str) -> Page:
    """Create documentation page"""

def update_page(project_id: str, page_id: str, content: str) -> Page:
    """Update page content"""

# Response format suggestion
{
    "id": "page-uuid",
    "name": "Agent Architecture Guidelines",
    "description": "Page description",
    "content": "# Markdown content here...",  # Preferred over HTML
    "content_html": "<p>HTML fallback</p>",
    "content_json": {},  # For rich editor content
    "created_at": "...",
    "updated_at": "...",
    "created_by": {...},
    "is_locked": false,
    "access": "public"
}
```

**Workarounds** (until implemented):
1. **Manual copy-paste** from Plane web UI
2. **Use Plane API directly** with HTTP requests
   ```python
   GET https://api.plane.so/api/v1/workspaces/{workspace}/projects/{project}/pages/
   ```
3. **Store docs in git repo** instead of Plane pages
4. **Build custom MCP extension** for pages (requires forking plane-mcp-server)

**Priority**: 🔴 **High** → Upgraded from Medium
- Originally thought "nice to have"
- Now confirmed as **completely missing** feature
- **Blocks knowledge management workflows**

- ❌ `expand="assignees,labels,state"`

- Trial and error to find working combinations
- Unclear which expansions are supported
- No documentation of valid expand values

**Suggested Fix**:
```python

  Supported values: assignees, labels, modules, cycles
  NOT supported: state (use state_name field instead)
```

## API Design Suggestions

### 1. Consistent Parameter Naming

**Issue**: Inconsistent naming across tools
- `project_id` vs `projectId`
- `work_item_id` vs `issue_id`

**Suggestion**: Pick one convention (snake_case) and stick to it

### 2. Better Error Messages

**Current**:
```
Input should be a valid string [type=string_type, ...]
```

**Better**:
```
Invalid state format. Expected state ID (string), but got state object.
Hint: Use expand="assignees,labels" instead of expand="state".
```

### 3. Tool Descriptions Should Include Examples

**Current**:
> "List work items in a project"

**Better**:
> "List work items in a project
>
> Example:
>   list_work_items(
>     project_id='abc-123',
>     expand='assignees,labels',
>     order_by='-updated_at'
>   )
>
> Supported expand: assignees, labels, modules, cycles
> Supported order_by: created_at, updated_at, priority, sequence_id
> Use '-' prefix for descending order"
