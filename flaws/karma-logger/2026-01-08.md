# Karma Logger Architectural Flaws

**Date:** 2026-01-08
**Reviewer:** Senior Agent Orchestrator
**Scope:** MVP_PLAN.md + DASHBOARD_MVP_PLAN.md implementation review

---

## Executive Summary

The karma-logger implementation follows a clean three-layer architecture (watcher → aggregator → persistence/display) but has **one critical integration gap** and several medium-priority inconsistencies that affect code maintainability and user experience.

**Overall Assessment:** 85% complete. Core backend is production-ready; TUI integration is broken.

---

## Critical Flaws (P0)

### FLAW-001: TUI Watch Mode Has No Data Connection

**Severity:** Critical
**Impact:** `karma watch --ui` renders but shows all zeros — completely non-functional
**Location:** `src/cli.ts:84-87`

**Current Code:**
```typescript
if (options.ui) {
  startTUI({ sessionId: options.project })  // Missing: aggregator!
  return
}
```

**Problem:**
- `startTUI()` receives `sessionId` only
- TUI's `useAggregator()` hook returns `null`
- `useMetrics()` returns `EMPTY_METRICS` (all zeros)
- No watcher created, no events flowing

**Expected:**
```typescript
if (options.ui) {
  const watcher = new LogWatcher({ projectPath: options.project })
  const aggregator = new MetricsAggregator()
  connectWatcherToAggregator(watcher, aggregator)
  watcher.watch()

  startTUI({ sessionId: options.project, aggregator })
  return
}
```

**Evidence:** Compare to working dashboard implementation at `cli.ts:152-167` which correctly wires all components.

## High Priority Flaws (P1)

### FLAW-002: Parser Logic Duplication

**Severity:** High
**Impact:** Inconsistent behavior between batch parsing and real-time watching
**Locations:**
- `src/parser.ts:61-92` (canonical)
- `src/watcher.ts:61-98` (duplicate)

**Problem:**
- `normalizeEntry()` implemented in both files
- Watcher's version is simplified, missing edge cases
- `extractToolCalls()` only exists in parser, not watcher
- Tool calls may be missed during real-time watching

**Symptoms:**
- Batch reports via `karma report` may show different tool counts than live `karma watch`
- Edge case handling diverges over time as one file is updated but not the other

**Fix:**
```typescript
// parser.ts - export shared functions
export { normalizeEntry, extractToolCalls, normalizeUsage }

// watcher.ts - import and use
import { normalizeEntry } from './parser.js'
```

---

### FLAW-003: Inconsistent Real-Time Update Patterns

**Severity:** High
**Impact:** 1-second display lag in TUI vs instant updates in other modes
**Locations:**
- `src/commands/watch.ts` — event-driven (correct)
- `src/dashboard/sse.ts` — event-driven (correct)
- `src/tui/hooks/useMetrics.ts` — polling (incorrect)

**Pattern Comparison:**

| Mode | Implementation | Latency |
|------|---------------|---------|
| Watch Streaming | `watcher.on('entry', ...)` | ~0ms |
| Dashboard SSE | `watcher.on('entry', ...)` → broadcast | ~0ms |
| TUI | `setInterval(getMetrics, 1000)` | ~1000ms |

**Problem:**
- `MetricsAggregator` only exposes `getTotals()` (polling interface)
- No `EventEmitter` pattern for subscribers
- TUI polls every second instead of reacting to changes

**Fix:**
```typescript
// aggregator.ts - add event emission
export class MetricsAggregator extends EventEmitter {
  processEntry(entry, session) {
    // ... existing logic ...
    this.emit('metrics:updated', this.getTotals())
  }
}

// tui/hooks/useMetrics.ts - subscribe to events
useEffect(() => {
  const handler = (metrics) => setMetrics(metrics)
  aggregator.on('metrics:updated', handler)
  return () => aggregator.off('metrics:updated', handler)
}, [aggregator])
```

### FLAW-004: No Automatic Persistence from Watch Mode

**Severity:** High
**Impact:** Live session data lost unless user manually runs `karma report --sync`
**Location:** `src/commands/watch.ts`

**Problem:**
- `WatchDisplay` shows live metrics but doesn't persist to SQLite
- Only `karma report --sync` calls `syncSessionsToDB()`
- Dashboard also doesn't auto-persist
- If user closes terminal, all watched data is lost

**Expected:**
- Periodic persistence (every 10-30 seconds)
- Or persist on graceful shutdown (SIGINT handler)
- Or background daemon mode with continuous sync

## Medium Priority Flaws (P2)

### FLAW-005: No Session Lifecycle Management

**Severity:** Medium
**Impact:** Memory grows unbounded; sessions never marked as "ended"
**Location:** `src/aggregator.ts`

**Problem:**
- `MetricsAggregator` accumulates sessions forever
- No `endSession(sessionId)` method
- `clear()` exists but is never called
- Dashboard shows stale sessions mixed with active

**Expected:**
```typescript
endSession(sessionId: string) {
  const session = this.sessions.get(sessionId)
  if (session) {
    session.endedAt = new Date()
    this.emit('session:ended', session)
    // Optionally persist here
  }
}
```

---

### FLAW-006: Activity Buffer is Ephemeral

**Severity:** Medium
**Impact:** Tool call activity history lost on terminal close
**Location:** `src/commands/watch.ts:42-64` (ActivityBuffer class)

**Problem:**
- Each `karma watch` creates fresh ring buffer
- Activity not stored in aggregator or database
- No way to replay recent activity after restart
- Multiple TUI instances can't share history

**Expected:**
- Activity flows through aggregator
- Optional persistence to database
- History replay on reconnect

---

### FLAW-007: Type Mismatch Between Memory and Database

**Severity:** Medium
**Impact:** Manual conversion required; potential data loss
**Locations:**
- `src/aggregator.ts` — `SessionMetrics` (in-memory)
- `src/db.ts` — `SessionRecord` (database)

**Differences:**

| Field | SessionMetrics | SessionRecord |
|-------|---------------|---------------|
| models | `Set<string>` | JSON string array |
| toolUsage | `Map<string, number>` | JSON object |
| cost | `CostBreakdown` object | 5 separate columns |

**Problem:**
- No defined converter function
- Conversion scattered across save/load methods
- Easy to miss a field during changes

**Fix:** Create explicit `toRecord()` and `fromRecord()` methods.

## Low Priority Flaws (P3)

### FLAW-008: No Pricing Override Configuration

**Severity:** Low
**Impact:** Enterprise customers with custom pricing can't get accurate costs
**Location:** `src/cost.ts:34-76`

**Problem:**
- Model pricing hardcoded
- No config file support for custom rates
- No way to account for volume discounts

---

### FLAW-009: Missing Database Foreign Key Constraints

**Severity:** Low
**Impact:** Orphaned agents possible if parent deleted
**Location:** `src/db.ts` schema

**Problem:**
- `agents.parent_id` has no FK constraint
- `agents.session_id` reference not enforced
- No cascade delete rules

---

### FLAW-010: Watcher Doesn't Handle Symlinks

**Severity:** Low
**Impact:** Projects using symlinked `.claude` directories not discovered
**Location:** `src/discovery.ts`

**Problem:**
- `findJsonlFiles()` uses `fs.readdir` without `followSymlinks`
- Edge case but affects some development setups
